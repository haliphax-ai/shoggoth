# 2026-05-09

## OpenRouter Video API Discovery

Investigated why OpenRouter video generation was returning 500 errors. Root cause: the `openai-video-async` adapter was hitting `POST /chat/completions` for video models, but OpenRouter uses a **completely separate video API**:

- **Endpoint:** `POST /api/v1/videos` (not `/chat/completions`)
- **Request body:** `{ model, prompt, duration?, resolution?, aspect_ratio?, frame_images? }`
- **Response:** `{ id, polling_url, status: "pending" }`
- **Poll:** `GET /api/v1/videos/{jobId}` → `{ status: "completed", unsigned_urls: [...] }`
- **Download:** `GET unsigned_urls[0]`

Status values: `pending` | `in_progress` | `completed` | `failed`

## Implementation Complete

- Adapter implemented via workflow (5 tasks, all passed after 1 retry on test task)
- Bug fix: adapter was sending empty `frame_images: []` and hardcoded `resolution: "1080p"` causing 500s. Fixed to only send optional fields when populated.
- Config fix: OpenRouter baseUrl needed to be `/api/v1` (not `/api`) since `normalizeBaseUrl` only appends `/v1` if missing.
- Verified end-to-end: `minimax/hailuo-2.3` generated a 6s video successfully via the new adapter.
- Note: Hailuo only supports 6s or 10s durations (not arbitrary).
- Config updated: all OpenRouter video models now have `"adapter": "openrouter-video"` explicitly set.
- Plan updated: no glob/pattern matching for model routing — explicit model names only, adapter set per-model or via provider `defaultAdapter`.
